1.   Introduction

What is RTSC?

Reusable Target Software Content is an “umbrella” name for an SDO initiative to deliver reusable target content (e.g., DSP/BIOS, xDAIS, Codec Engine, etc.) and foundational tools that enable developer’s of embedded real-time software to develop new components, deliver components in binary (or source) form to consumers, and to re-use components developed by other developers.

What are XDCtools and Iliad?

XDCtools is the name for the foundational tools product that is part of the RTSC initiative.  These tools provide the base tools required to create and use components.  Iliad is the internal engineering project name for the XDC tools development effort.  

What do the letters XDC stand for?

XDC originally stood for eXpress DSP Components, but now that the tools and the component model are being applied to all embedded content (not just DSP) and support non-TI tool-chains, XDC stands for eXpanDed C.

The latest internal engineering release is always available from the Iliad home page.  One can also review changes between any two successive releases from this page.  All external production releases of the XDCtools are available from here.

What documentation is available?

The current documentation is summarized in the table above.

Can I download a previous release?

Previous weekly releases are available from the TISB Release Download Site.

If you have access to the TISB development servers and your workstation is configured to use the TISB tools, it is possible to get any previous release. To install the Windows release to the directory ./xdc type:

z:/library/trees/xdc-rel/src/iliad/install.ksh ./xdc

where, z: is a network drive for \\Sambasb\ztree and rel is the release you want to install.  The install.ksh script will install Solaris and Linux releases if you pass the ‘-u’ or the ‘-l’ options, respectively.

The XDCtools supported on most Windows platforms and x86 platforms running Red Hat Linux release 7.2 or above.

The production releases of XDCtools are “wrapped” in an installer that must be run from your development workstation.  The installer can be run via the command line so it is possible to run the installer from a tenet session, for example.

To view existing bugs, you must first log into the web page above using the account “readonly”, password “readonly”.  You can then locate RTSC bugs using saved queries under the “Public Queries/Project Queries/RTSC” folder on the left.  If you are not in the SDO organization, send bug reports to me and I'll file them on your behalf.

4.   Component Model verses …

What is a Component?

The term “component” is overused and therefore loses much of its value in technical conversations.  However, many in the software industry are adopting Clemens Szyperski’s definition (from “Component Software, Beyond Object-Oriented Programming”, 1998):

Software components are binary units of independent production, acquisition, and deployment that interact to form a functioning system.

Szyperski also describes a definition that was first formulated at the 1996 European Conference on Object Oriented Programming (ECOOP) as an outcome of the Workshop on Component-Oriented Programming.

A software component is a unit of composition with contractually specified interfaces and explicit context dependencies only.  A software component can be deployed independently and is subject to composition by third parties.

Using these definitions, we see that components differ from classes in object-oriented systems in that components are units of deployment; most classes in an object-oriented system cannot be independently delivered into and reused in a different application.

Independent of any definition, Szyperski and Messerschmitt [Szyperski 2003] identify five essential properties of components: “a component should be multiple-use, non-context-specific, composable with other components, encapsulated (cannot be modified or, typically, even examined) and a unit of independent deployment and versioning.”

What problem do components solve?

Component models exist to facilitate binary (and source) software reuse among development groups in potentially different companies; i.e., wide-scale reuse of software.  In a recent study, the most important positive factor in software development productivity is “reuse of high-quality deliverables” [Jones 2000].

Because components are independently deployable they are also independently replaceable in a system.  Thus, components also help manage the impact of changes to other elements of a system; e.g., new hardware platforms, new implementations of standard algorithms, new communication peripherals, etc.  In fact, Cheesman and Daniels [Chessman 2001] claim this is the primary benefit of components.

How are components different from libraries of pre-compiled code?

Component models define standards that allow reusable software elements to be created, installed, and used with minimal user interaction (other than to select the components of interest).

Libraries cannot be used without headers that define the interfaces provided by the modules contained in the library.  So, at a minimum both libraries and headers must be bundled into a “package” for delivery.  Components are units of delivery.  Thus, component models define how these items are bundled so that tools (such as IDE’s and build tools) can automatically create, install, and use components from multiple suppliers within a single development environment.

Libraries do not contain “meta-information” to allow them to be used without extensive documentation and ad-hoc installation rules; e.g., are the modules in the library compatible with big-endian or little-endian memory systems, have they been compiled to assume near or far data memory or extended program memory, is there a “debug” version of the library and is it compatible with other non-debug components?  Components must be inter-operable.  Thus, component models define where meta-information is placed and what meta-information is required so that tools will never allow incompatible combinations to be created.

Libraries can only support a single target; all the modules in a library are built for a single ISA and a single run-time model.  In contrast, components must support multiple targets and multiple execution platforms; e.g., a “tone detector” component might support both a C5000 and C6000 target (each with a variety of run-time models) with the same programmatic interface.

What other properties of components make them more re-usable?

Components implement multiple interfaces.  This allows them to take an active role in logically separate phases of a system’s lifetime; e.g., component models often require the implementation of “discovery” interfaces that allow components to self-identify one another and establish “bindings” between themselves and other components without the application being involved.  In this way, components can be “self-configuring”, “self-checking”, and “self-installing”.  By allowing the component to actively participate in the assembly of the system, the system integrator is no longer required to sort through component specific documentation to ensure that complex dependencies are satisfied.

Suppose, for example, that some component A only works with a specific set of codecs, relies on a specific revision of the DSP (because earlier DSP revisions had bugs that would require expensive software workarounds), and depends on a specific release of the DSP/BIOS or CSL.  When component A is configured into a program, design-time interfaces provided by component A will be activated and A’s implementation of these interfaces can verify all of A’s requirements.  Therefore, many of today’s common integration problems can be reliably detected and reported before running the program. 

Is xDAIS is a component model?

Yes.  However, it has two significant (self-imposed) limitations:

1.      xDAIS does not provide (or require) any tooling beyond the C compiler to assist in the creation or consumption of xDAIS components, and

2.      xDAIS specifically limits its scope to algorithms (“pure mathematical transformations with persistent state”).

RTSC, in contrast, provides tools that allow a language independent specification of interfaces (including the interfaces defined by xDAIS), build tools that facilitate the creation, packaging, installation, and use of components.  In addition, RTSC does not limit its applicability to just DSP algorithms; for example, it is equally useful for TCP/IP stacks and device drivers as for algorithms.

In summary, RTSC broadens the scope of xDAIS and provides tools that facilitate the creation and use of components.

5.   RTSC verses …

Why not C++, embedded C++, Java, …?

C++, Java, and embedded C++ are examples of languages not of component models.  A component model defines additional structure on software developed in these languages necessary to support interoperability among components.  Languages by themselves are not sufficient to ensure interoperability.  In addition, the objects that object-oriented languages support are often too low-level for effective reuse [Long, 2001].

While some component models (e.g., JavaBeans) have a preferred language as part of the model, many component models (e.g., COM) support the implementation of components using a variety of disparate languages.

Why not COM, CORBA, JavaBeans, …?

Most component models are designed to support heterogeneous distributed workstation computing environments.  These component models are poorly suited for memory constrained embedded systems nor do they work well in a real-time environment.  Moreover, because these models are intended to support web-based applications, they must address a number of issues that are simply not necessary in a deeply embedded system; serialization, uniform data transfers and event notifications, security, database integration, compound document support, etc.

In addition, these component models do not address the need for a single component to support multiple targets.  Imagine a codec driver that supports the DSK6201, DSK6211, and a DSK64xx platform where each platform has incompatible DMA, serial port, or codec peripherals.  Some components may even span ISAs; e.g., a tone detector may be developed and delivered to either a C6000 or a C5000 system.

Does anyone else have a component model for embedded systems?

Yes.  Several component models are designed specifically to address the needs of embedded systems.  However, most of these models are research prototypes and are not widely used.  The models that are most closely aligned with the requirements of resource-constrained devices include Koala, ECOS Component Model, Knit, PECOS, and ADI’s VisualDSP Component Software Engineering (VCSE) tools that are part of Release 3.0 of VisualDSP++.

Koala is a component model developed at Philips to facilitate component re-use among their consumer electronics divisions.  Thus, the model has minimal footprint, can be used in real-time environments, and supports re-use among widely separated development organizations.  Like RTSC, Koala supports design-time instantiation of components.  However, Koala’s “diversity interfaces” (the equivalent of RTSC configuration interfaces), require that source code for the component be provided.

The ECOS Component Model is a component model designed to enable extensions to Redhat’s ECOS embedded real-time kernel; e.g., TCP/IP stacks, ITRON and POSIX personality layers, drivers, etc.  This model also meets many of the performance requirements of DSP systems.  However, it does not allow static instantiation of components in a system and it requires components to be delivered in source form.

Neither ECOS nor Koala has any deep integration with the compiler.  This limits their ability to do global optimizations that eliminate the overhead associated with the component model.

Knit is a University of Utah, School of Computing research project.  This component model does attempt to perform global optimizations by providing all component sources to the GNU C compiler, which then is able to do cross component optimizations.  Thus, like the ECOS model, Knit requires components to be delivered in source form.

PECOS is a European research project that attempts to address many of the same problems as RTSC.  However, this project is not yet at the point where it can address the problems already handled by RTSC.  Although it is unlikely that a product can be made from this technology within the next few years, this project should be watched closely.

Although VSCE is intended for embedded DSP applications, it differs from all other embedded approaches and follows Microsoft’s COM model very closely.  There is no notion of design-time instantiation, all components are created at runtime via “factory” interfaces, and all components are called from C++ clients.  In other words, they’ve not attempted to address the unique non-functional requirements of embedded software.

This approach, while proven to work in a Windows environment, limits the applicability of componentization to fairly “heavyweight” elements of a system; since the overhead of getting into and out of a component is relatively large, the component itself must consume significant resources.  ADI may be trying to “shoot ahead of the duck” by assuming that by the time components become popular in embedded DSP systems, Moore’s Law will make the overhead of VSCE insignificant.  On the other hand, ADI is relying on an approach that Microsoft itself is replacing; creating components by directly writing COM objects in C++ is a dying art.

In contrast, the RTSC component model is suitable for all reusable target content; not just computationally intensive DSP algorithms.

How is RTSC similar to other component models?

RTSC shares the following characteristics with virtually all other component models:

1.      components may implement multiple interfaces and the same interface is often implemented by more than one component in the same program,

2.      components include “meta-information” that can be used by other components as well as tools that facilitate the integration of components into a program, and

3.      components consist of a diverse set of files that are bundled or “packaged” using a tool specified by the component model.

All component models provide a mechanism for components to present multiple interfaces to client applications.  This simple idea makes it possible for a single component to

1.      be effortlessly integrated into a variety of systems each defining a required interface,

2.      simultaneously support different (incompatible) versions of the same interface, and

3.      implement multiple interfaces that represent very different aspects of the component within a system; e.g., a diagnostic interface that allows real-time interaction with the component to view internal activity and report non-fatal error conditions, a quality of service interface that allows real-time tradeoffs between MIPS or power and audio or video quality, and the component’s “primary” interface that performs some component specific function.

An Interface Definition Language (IDL) tool is used to define RTSC interfaces.  The IDL language is proprietary but like all other IDL languages it follows the C/C++ syntax for declaration of interface methods and types (structures, enums, …).  COM and CORBA followed this approach because it allows one to implement components in any language.  Thus, it makes it possible to easily re-use an existing base of C/C++ or assembly code (a requirement for DSP) and compatibly support new languages.

Sun’s JavaBeans and Microsoft’s .NET took a different approach; new implementation languages were created (Java and C#) that incorporate the functionality of the IDL into the language.  While this approach is much cleaner from a computer science point of view, it requires the developer community to learn a new language and re-write the existing code base.

All component models support meta information (introspection and reflection) and a component transfer format.  Meta information allows “framework” software to manipulate disparate collections of components; e.g., a framework may need to determine if a component implements an optional interface (either at design-time or run-time) and if it does, then the framework uses the interface.  A component transfer format is simply a means to collect all files necessary to use a component into a single file; e.g., Java uses jar.

RTSC supports meta information and its component transfer format is currently a tar file.

How is RTSC different from other component models?

While all component models provide a means for a component to declare and implement multiple interfaces, in RTSC, we also allow declaration and implementation of interfaces that do not run on the targeted platform.  These interfaces are invoked at design-time in the host development environment.  This allows components to configure themselves and even select different implementations of their interfaces based on user specified requirements, other components in the system, and even the particular platform that executes the component’s code.  Thus, RTSC components execute in two very different domains: the host development environment and on the embedded platform.

In addition, RTSC interfaces allow developers to declare “configuration parameters” which can be set by the client and control aspects of the component’s implementation or runtime behavior.  These parameters are validated at design-time, which allows the run-time code to be as minimal as possible; little or no runtime error checking is necessary.  In the future, the Alchemy compiler will be able to treat these parameters as manifest constants in the source and eliminate (via symbolic execution and sophisticated constant folding) large portions of the run-time code.

Without these capabilities, it is virtually impossible to use a component in more than just a few systems, defeating the very purpose of a component technology: re-use.

Every component model defines some mechanism for packaging the component for delivery.  In order to create the package, the component producer must enumerate all the files that are to be included; either in a makefile or an explicit “manifest” file read by a packaging tool.  The manifest file in RTSC not only describes what is to be included in the package but it also drives a “build engine” that makes the creation and maintenance of packages as simple as possible.

Finally, RTSC is the only embedded component model that supports binary delivery.  Although it is possible to provide component in source form using RTSC, there is virtually no performance benefit to doing so.

If design-time interfaces and configuration are so valuable, why aren’t they part of mainstream component models?

In mainstream component models (COM, JavaBeans, CORBA, etc.), there is very little variation in the operating environment seen by the component; the execution platforms share many common base capabilities (large uniform memory pools, network and file services, display and input devices) and low level software that is affected by hardware variations is not delivered as components.  Moreover, there are very few “non-functional” requirements placed on the components themselves.  For example, the size of the component and its execution time are far less important than its function in the system; if a component is large and slow it can still be widely deployed and improved over time.  These factors allow a single component implementation to effectively operate on a variety of platforms.

In contrast, components for embedded systems include device drivers and must support literally thousands of radically differing platforms; each customer creates a custom platform with a unique set of peripherals, a custom memory map, and a particular version of a DSP.  In addition, each platform has numerous “non-functional” requirements, which also affects the implementation of a component; e.g., real-time constraints, memory footprint limitations, and even power limitations.  Optimization of a G.729 vocoder for minimal footprint often conflicts with optimization for minimum execution time, for example.

Thus, within the embedded DSP space of applications, it must be possible for a component to support multiple implementations.  RTSC components implement design-time interfaces that allow the component to automatically select the optimal implementation of itself for the system being constructed.  Moreover, components can define “configuration parameters” that allow the client to further customize the component for the application being developed.

The configurability provided by RTSC allows producers to create components that can be customized by the consumer to meet stringent “non-functional” operating constraints (MIPS, memory, power, real-time, etc.) without the producer ever having to directly engage the consumer or deliver a component’s source code.

Do other embedded component models support design-time configuration?

Yes.  Virtually every embedded component model (except ADI’s VSCE) supports design-time configuration and the model’s documentation stresses it importance.  However, in every case, design-time configuration is achieved by requiring components to be provided in source form.  Even Koala, which borrows heavily from the binary model COM, requires source level components to support its “diversity interfaces”.

What is the relationship between xDAIS and RTSC?

xDAIS defines a component model by defining a specific and highly disciplined use of the C language.  Interfaces are specified by creating C/C++ headers that obey numerous requirements detailed in the xDAIS Specification.  In addition, xDAIS defines an interface (IALG) that must be implemented by all algorithms.

In contrast, RTSC provides the tools that enable one define new interfaces in a language neutral manner; via the RTSC Component Definition Language (CDL).  By defining interfaces using this CDL, it is no longer necessary to manually review an arbitrary C header to ensure that the rules are being adhered to, the runtime model is no longer explicitly visible in the interface definition, and we can support arbitrary component implementation languages (C, asm, C++, embedded C, Java (!), etc.)

In addition, RTSC does not define any interfaces that must be implemented by target modules.  It does define a number of design-time interfaces that must be implemented by those adding new targets (C6000 big endian, C5500 far model, etc.) and new platforms (DSK6211, EVM5409, etc.).   However, no restriction is placed on the runtime interfaces defined by target modules.

Is RTSC a replacement for xDAIS?

No.  RTSC defines a common set of requirements for all components and the XDCtools supplies a set of “foundational tools” that facilitate the creation and consumption of components.  However, RTSC does not impose any restrictions on the interfaces supplied by components or on the non-functional aspects of components (maximum interrupt latency, reentrancy, relocatability, code footprint, stack usage, etc.).  In other words, it is still possible to create a bad component using RTSC.

The same study that concluded that the most important positive factor in programmer productivity is the reuse of components also concluded that the most negative factor in productivity was “reuse of poor-quality deliverables” [Jones 2000].  RTSC enables reuse of components while xDAIS ensures that the components are of high quality.

6.   Creating RTSC Components

What do I need to learn to create a RTSC component?

The answer depends on what features of the RTSC component model you want to use.  RTSC has comprehensive support for legacy code bases, which allows one to convert an existing library into a RTSC package literally within minutes; no changes to the legacy sources are required.  On the other hand, to take full advantage of the RTSC component model one must learn the syntax of the RTSC IDL, the basics of JavaScript syntax, and the RTSC build model (expressed in the RTSC IDL). 

RTSC is designed to ensure that, even when taking full advantage of its capabilities, these requirements are kept to an absolute minimum.

Why not use XML instead of custom IDL to specify a RTSC component?

Two reasons: XML defines syntax not semantics and XML makes a very poor human interface.  XML by itself is not sufficient to specify components; we still need to define the semantics of RTSC components and these semantics will not be better understood by tools if the component specifications conform to XML syntax.  Moreover, specifications written in XML are harder to write and review by the very developer’s we are trying to help to create components.  Terence Parr, Chief scientist at jGuru.com has argued convincingly [Parr 2001] that “XML makes a lousy human interface.”

“Humans have an innate ability to apply structure to a stream of characters (sentences), therefore, adding markup symbols can only make it harder for us to read and more laborious to type. … Rather than spending the time to design and parse a human-friendly language, programmers are using the fastest path to providing a specification language and implementation: "Oh, use XML. Done." And that's OK, but I want programmers to recognize that they are providing an inferior interface when they take that easy route.”

On the other hand, it is important to enable third-party tools to read and react to component specifications without requiring a proprietary IDL parser.  So, during creation of a RTSC package, all IDL specifications are “compiled” into XML files that are included in all releases of the component.  In addition, both Java and JavaScript interfaces are provided to enable components to easily reflect on any RTSC component specification.

Can existing libraries (non-RTSC) be used without having to re-build them?

Yes.

Can existing xDAIS components be used without modification?

Yes.

Do I have to use (or learn) C++?

No.  Although early versions of the RTSC IDL only had bindings to C++, it currently supports a C binding; so, no knowledge of C++ is required.

Do I have to use (or learn) Java?

No.  Although some of the XDCtools tools are implemented in Java, the user of the XDCtools does not need to know (or drink) Java.

Do I have to use (or learn) JavaScript?

RTSC allows components to be “configured” by the consumer of the component.  The configuration language used by RTSC is ECMA-262 (aka JavaScript).  Note that JavaScript has no relation to Java.  JavaScript is a scripting language invented by Netscape and has nothing to do with Sun’s Java technology.

Therefore, some familiarity with JavaScript is helpful.  But because configuration scripts typically consists of simple assignments to variables and the JavaScript syntax is very close to the C language, the embedded C programmer will have no difficulty reading or creating RTSC configuration scripts.

If you have used DSP/BIOS configuration scripts, you already have enough JavaScript background; the DSP/BIOS configuration tool builds atop the RTSC configuration engine.

Why not use Perl (TCL, Python, …) instead of JavaScript?

We considered several alternatives to ECMA-262 (a.k.a JavaScript) including Perl, TCL, Python, Ruby, VB, and the creation a proprietary language.  We chose ECMA-262 because:

  • it is a standard language (http://www.ecma-international.org/publications/standards/ECMA-262.HTM),
  • ECMA-262 has numerous excellent books and reference manuals ([Flanagan 2002]),
  • its syntax is virtually identical to C; our experience is that C programmers can read scripts with *very* little training or documentation,
  • existing DSP/BIOS applications use this language to configure the BIOS,
  • there are numerous freely available cross-platform OS-independent implementations in a variety of languages (we are currently using a freely available interpreter written in Java from Mozilla but may switch to Mozilla’s C++ implementation in the future), and
  • the implementation we’ve chosen is actively supported, used in a variety of other projects, and includes a cross-platform debugger.

Where can I get a good JavaScript reference?

The best reference book is the O’Reilly JavaScript book ([Flanagan 2002]).  This book is very well written and cleanly separates the description of the JavaScript language from its use in Web browsers and the browser’s Document Object Model (DOM).

Although both Microsoft and Netscape publish JavaScript references one must be careful to avoid proprietary extensions that are not part of the ECMA-262 specification.  If you need an online reference, Netscape’s Core JavaScript Reference is one of the best.  Netscape’s Core JavaScript Guide, however, contains non-ECMA-262 compliant extensions.

The final word, however, is the language ECMA-262 specification available from the ECMA web site; http://www.ecma-international.org/publications/standards/ECMA-262.HTM.

I already have a CCS project, how do I make this into a RTSC package?

Adapting an existing project into a RTSC package is quite straightforward; no changes are necessary to the source code is required.  Creating a RTSC package from legacy content only involves the addition of a few new files that describe the package.

The first file that must be created is a “build script”, package.bld.  This build script describes the contents of the package; what targets are supported, what libraries are to be created, what platforms are supported, which objects go into each library, what headers are necessary for the client of the package, etc.

The second file is a package configuration file, package.xs.  This file defines a function that is called to determine the name of the library to use when linking a program that uses your package; each package is responsible for automatically selecting the appropriate library for the program being built.  This function is passed information about the platform that the program is being targeted for as well as the compilation options for the program.  Thus, a single package may transparently support multiple platforms and runtime models without requiring the user of the package to know the names of the various libraries included in the package.

Finally, you must create a package definition file, package.xdc.  This file declares the name of the package and all modules and interfaces in contained in the package.  This file also contains package compatibility information and package-level documentation.

The online examples contain several examples of legacy code bases that are “re-packaged” as RTSC components; see the examples in the examples/legacy sub-directory.

Does RTSC require that all algorithms be xDAIS compliant?

No.  You can easily create algorithms that are not xDAIS compliant (although it is politically incorrect to do so.)

7.   Using RTSC Components

What is the overhead associated with RTSC?

In the worst case, the execution time overhead is equivalent to the execution time overhead of xDAIS algorithms; i.e., there is a single function call indirection between the caller and the component’s code.  Since RTSC supports legacy code bases, the overhead can be eliminated by simply avoiding certain features of RTSC modules.  However, we are actively working with the compiler team’s Alchemy technology to ensure that by the time Alchemy is available we can eliminate this already very low overhead.

The footprint of RTSC is also very small.  For example, no runtime memory allocation is required.  Thus, like xDAIS, it is possible to create a static system with a footprint virtually identical to the footprint of a “traditional” non-component based application.

What do I need to learn to use RTSC components?

If you are already familiar with the DSP/BIOS configuration mythology and the textual configuration tool, then you already know how to configure components and build them into your application.  Calling methods defined by a component from your application is as easy as calling methods in a C or C++ library (which, of course, depends on the interfaces provided by the component).

The language understood by the textual configuration tool is ECMA-262 Revision 3 (aka JavaScript).  However, because configuration scripts consist primarily of assignment statements, if-then-else blocks, and simple loops, and because JavaScript syntax matches C, the embedded programmer already knows enough JavaScript to write basic configuration scripts.

Why is the RTSC configuration language JavaScript?

The RTSC development team had to choose between inventing a proprietary configuration language or choosing a standard language for which there are existing textbooks.  We decided to use a standard language. 

Several standard languages would have worked; e.g., TCL, Python, Perl, JavaScript, Ruby, and VBScript.  We chose JavaScript because it’s widely used and documented and its syntax is so similar to C that any embedded developer will be able to read and modify (correctly) a JavaScript script.  Moreover, JavaScript is freely available on virtually all host development platforms (unlike VBScript).

Finally, it is important to note that the architecture of modern scripting languages provides a good separation between the language and the objects being manipulated by the language.  Thus, it is possible to use alternative script languages to manipulate the configuration objects in the future.  On the other hand, doing so makes the support and documentation burden somewhat larger.

Do I have to re-write existing software to take advantage of RTSC?

No.
