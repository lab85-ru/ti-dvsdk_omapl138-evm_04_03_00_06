1.   Introduction

What is the XDC Build Engine?

The XDC Build Engine consists of command line tools and an object model that enable developer’s to manage the build, deployment, and use of the files that makeup an XDC package.  These tools allow one to build, deploy, and use content in exactly the same way for any target (C6000, C5000, C2800, ARM, MIPS, …) executing on any platform (Evm, Dsk, TEB, custom HW, …) from any development host (Windows PC, Linux PC, Solaris Sparc).

What problem does it solve?

The embedded developer’s job is characterized by multiplicity; multiple embedded processors (ARM, MIPS, C6000, C5000, C5500, C2800, …), multiple tool-chains  required support these processors (GNU, Microsoft, ADS, TI, …), multiple run-time models (big verses little endian, near verses far data), and multiple end-equipment platforms that run executables.  This contrasts and often conflicts with the practical need to maintain a single reusable (i.e., portable) code base – one of the embedded developer’s most powerful productivity assets.

The XDC Build Engine greatly simplifies the production and maintenance of libraries and executables originating from portable source code bases by allowing developers to concentrate on what to build rather than on how to build their embedded software.  It does this by encapsulating the variations required to build code for multiple targets using any development host and run the code on any platform.  This encapsulation enables tool-chain experts to define new targets, board manufactures to create new platforms, and embedded software developers to build for these targets and platforms by simply naming the targets and platforms that their software supports.  Being host independent, the XDC Build Engine enables developers with different preferred development hosts to transparently share (re)buildable code bases.

Creation of an executable that runs on an embedded platform requires use of a particular tool-chain (compiler/assembler, linker, archiver) with a specific set of options to ensure compatibility with libraries produced by, say, algorithm producers.  In addition, for performance reasons, most embedded platforms require that executables be linked against a platform specific memory map.  These platform and tool-chain dependencies make it difficult to manage portable code bases that, by definition, are supposed to be independent of these specifics.

 The XDC Build Engine eliminates these dependencies by encapsulating them into “targets” and “platforms” which are produced and maintained outside the package containing the reusable embedded software.  This separation of concerns allows embedded developers to focus on what to build rather then on how to build their software.

How does the XDC Build Engine help me manage libraries?

Creation of a reusable libraries requires specification of the object module file format (COFF, ELF, PE, …), a compiler runtime model (big verses little endian, near verses far data, etc.), and an instruction set (C6x, C548, C5402, ARM, MIPS …).  All of these attributes are encapsulated in the XDC Build Model’s definition of a “target”.  Using the XDC Build Engine, it is possible to build a library by simply naming the set of targets that the library supports and (of course) the source files that contribute to the library.  The XDC Build Engine takes care of building the library for each target the package specifies.  The resulting package, therefore, supports any client requiring one of the targets specified; e.g., C62x big endian small model, C62x little endian large model, etc.

When a new target becomes available, the developer simply declares (in the package’s build script) that the library supports the new target and re-builds.  If a package is distributed with its sources, the consumer can trivially re-build the package for a different set of targets using any development host.

I’m only using a single DSP, how do I benefit from multiple target support?

Most embedded programmers that use a DSP must develop for at least two incompatible CPU architectures; one or more DSPs that perform real-time signal processing and at least one General Purpose Processor (GPP) that serves as a system controller.  In addition, many embedded developers also maintain a “golden C” version of portions of their software which is compiled for execution on their development workstation.  Algorithm developers, for example, maintain such a version to make it easier to develop and debug enhancements of their algorithms; development of the algorithm can proceed without hardware, without the resource constraints inherent in high-volume low-cost platforms, and use sophisticated development tools such as Rational’s Purify and Compuware’s BoundsChecker.  The “golden C” version also provides a means for validating the correctness of each port of the algorithm to a new platform.

Thus, it is common practice for portable C source files, that are part of a single system, to be compiled using three disparate compiler tool chains: the DSP compiler (from TI, ADI, or DSP Group), the GPP compiler (from FSF, ADS, or Intel), and the native workstation compiler (from Microsoft, SUN, or FSF).

How does the XDC Build Engine help me validate my code?

It is sometimes said that, “there is no such thing as portable C code, only C code that has been ported.”  Simply running a code base through disparate compilers (e.g., Microsoft’s cl, GNU’s gcc, and TI’s cl) that employ various degrees of static code analysis can improve a portable code base.

In addition, each Build Engine target supports multiple “profiles”.  A profile is a named combination of tool-chain options that uses the tool-chain to produce, say, debug libraries, optimized libraries, libraries that support code coverage measurements, libraries that support execution time profiling, etc.  By making these profiles available to the developer in a manner that is easy to use and independent of the tool-chain, it encourages experimentation and use by the developer.

The XDC Build Engine allows the developer to use best-in-class features from a variety of tool-chain suppliers.  For example, gcc supports code coverage measurement but TI compilers don’t.  Using the XDC Build Engine it is trivial to build the same code base using gcc with the “code coverage” profile and TI using “optimized for execution speed” profile.  By running test vectors through the gcc version one can gather code coverage data.  Then using the same test vectors verify functional correctness of the TI optimized for time library.

Finally, the XDC Build Engine has built-in support for creating and running collections of tests.  This makes it easy to create, run, and even deliver a (re)buildable regression test suite.

What is a build script?

Build scripts can be thought of as a high-level equivalent of makefiles in a traditional embedded build environment.  Every XDC package has a build script named package.bld that specifies what should be built and what files are included in a package.  For example, a build script might specify that a particular library must be built for all C62 targets (big endian, little endian, small model, large model, etc.) and that the file README.txt be included in the released package.

Unlike makefiles, however, XDC build scripts are written in the JavaScript Language (ECMA-262) and specify what files to build rather than how to build the files.  This makes XDC build scripts very easy to create and maintain.

What are the “Pkg” and “Build” objects that appear in build scripts?

All package build scripts reference two distinguished global variables; Build and Pkg.  These variables are distinguished “entry points” to the Build Object Model.  This model is fully defined using the XDC Interface Definition Language, which makes the model type-safe and self-documenting.

The Build variable is an alias for the configurable module xdc.bld.BuildEnvironment, and Pkg is an alias for the configurable module xdc.bld.PackageContents.  A complete definition of these objects can be obtained by reading the XDC interface definition files for these modules; the bld package is contained in the “packages” repository of the XDC distribution.

The xdc.bld.BuildEnvironment module encapsulates the context or environment in which the build scripts runs; e.g., the set of targets and platforms to be used, the development host name, etc.  Build scripts use the information contained in Build to “populate” Pkg with libraries, objects, executables, and releases via the methods defined by xdc.bld.PackageContents.

What is the XDC “Build Object Model”?

The XDC Build Engine roughly follows the Model-View-Controller paradigm [Hunt 2000].  The XDC Build Object Model (BOM) is the abstract data model in this design. In this case, the model represents the collection of files to be built, the sources used to build the files, the targets that libraries are built for, and the platforms that run the executables produced by the build.

This design pattern allows multiple clients of the model to work together (by agreeing on the model) without having to know about one another and new clients to be added without having to modify existing clients.  Clients of the Build Object Model include packages that define new platforms, packages that define new targets, and programs (or scripts) that generate files necessary to use a particular compiler, debugger, loader, or profiler.

What are the “xdc” and “xdc.om” objects that appear in build scripts?

In addition to the two distinguished global variables, Build and Pkg, that are aliases for the top-level objects in the BOM, build scripts can reference another global variable: xdc.

The xdc object is, in effect, a namespace for top-level functions and variables useful for both build and configuration scripts.  The function for loading a module is “xdc.useModule”, for example.  By placing these functions under the xdc object we minimize the possibility of conflict with variable names that appear in user supplied scripts.

xdc.om is a distinguished object that is the “root” of all loaded packages; all loaded packages and their modules and interfaces can be directly referenced through this variable. For example, if the package “ti.targets” contains a module named “C62” and this package has been loaded, the C62 module can be referenced via “xdc.om["ti.targets.C62"]”.

What is the difference between a target and a platform?

Libraries are built for a target whereas executables are built for (and run on) a platform.

In the XDC Build Object Model, a target encapsulates all the information necessary to build an object file and archive it in a library.  Intuitively a target corresponds to a CPU Instruction Set Architecture (ISA) that is the target instruction set of the compiler together with one of the compiler’s supported runtime models (big endian, little endian, near, far, large, small, …).  A platform, on the other hand, encapsulates the information necessary to create, load, and run executables (i.e., fully bound programs).  Platforms include memory map information, external peripheral information, one or more CPUs and their revision numbers, as well as one or more boards and their revision numbers.

Executables are built to run on platforms and an executable can run on just one CPU of a particular platform.  Libraries and object files are built for a target and can be used in any executable that runs on a platform that supports the specified target.  Libraries and objects built for the same target may always be compatibly linked together into a single program.

Why is it important to distinguish between a target and a platform?

By separating the target from the platform, tool-chain producers can release new tool-chains independently of hardware manufacturers releasing new execution platforms.  Software developers can release object code libraries that support a set of targets without being tied to a particular execution platform.  Portable programs, such as algorithm library regression test suites, can be built to run on a new platform by simply naming the new platform and rebuilding.

What is the difference between a program and an executable?

A program is an abstract entity whereas an executable is a concrete file that can be loaded and run on a specific platform.  When we say that the “hello world” program runs on the DSK5416 and DSK6416 platforms, we really mean that the same “hello world” program sources are used to build distinct executables (one per platform) that, when run on the corresponding DSK platform, display the words “hello world”.  Platforms run executables and these executables are concrete realizations of abstract programs.

What embedded targets are supported?

Support for the TI C6000, C5000, and C2800 families of processors is included in the XDC distribution.  However, the build model enables one to easily add support for virtually any other processor and tool-chain.

What are native targets and platforms?

Native targets represent the development host’s CPU and tool-chain that runs on that CPU.  Similarly, the native platform is the host development system’s platform.  For example, if you are using the Windows distribution of XDC, the native platform is the PC (represented by the host.PC package) and the default native target encapsulates Microsoft’s C/C++ compiler for the PC (represented by the microsoft.targets.Win32 module).

What native targets and platforms are supported?

Native targets and platforms exist for all development hosts that run the XDC tools (Solaris Sparc, Linux x86, and Microsoft Windows x86).  Selected GCC targets support a variety of Solaris and Linux development hosts.

What development hosts are supported?

The CDK is supported on PC platforms running Microsoft WinNT, Win2K, and XP, or Red Hat Linux release 7.2 or above Linux, and SUN Sparc platforms running SUN OS 5.7 or above.

What documentation is available?

Because the XDC Build Object Model is defined using the XDC interface definition language, reference documentation of all of the methods and parameters is available by simply reading the *.xdc files in the bld package.  In addition, a preliminary set of training slides (doc/build.ppt) is also located in the bld package.

The bld package is part of the XDC distribution and can be found by looking in xdcroot/packages/bld, where xdcroot is the installation directory of the XDC distribution.

Documentation of the xdc command (used to build packages and run test suites) is available in xdcroot/doc/xdc.html as well as on-line.

How do I report bugs?

Bugs should be reported against the XDC Component Developer’s Kit (CDK).  Please see the CDK FAQ for detailed instructions.

2.   Legacy Code Support

Can I use the XDC tools to build legacy programs/libraries?

Yes.  The examples in xdcroot/examples/legacy show how to do this.

Do I need to change any of my sources to use XDC?

No.

Do users of my XDC package need to use the XDC build tools?

No.  However, there are significant advantages for those that use these tools; e.g., it is easier for client programs to ensure that the right set of libraries is being used.

Do I need to make any changes to make my legacy deliverables an XDC package?

Yes.  The XDC Build Engine requires that all packages conform to a particular physical design.  It is this uniformity across all packages enables the XDC tools to operate at higher level than, say, make. 

The package producer has considerable flexibility in the physical design of the package, however.  Beyond requiring that all of a package’s files must be in a package directory (or sub-directory of this directory) that matches the name of the package and that all interface headers be in this package directory, the XDC Build Engine physical design requirements do not significantly constrain the internal organization of the package’s files.

Finally, each XDC package must include the file package.xdc which names the package, declares its compatibility with previous versions, and declares any dependencies on other XDC packages.  The following is a typical package.xdc file for a legacy code base that implements a G711 codec.

package g711 {
};

My application requires its own linker command file, do I need to change it?

No.  The XDC Build Engine gives the developer complete control over the linker; for more information about how to do this, see the “Creating Executables and Running Tests” section of this FAQ.

Tying a program to a TI linker command file implies that the program will only run on a platform supported by TI targets; i.e., the program can only be ported to platforms that support TI targets.  In many cases this is perfectly acceptable (especially from a TIer’s point of view).  However, in order to support programs that are portable to non-TI platforms (e.g., x86-Win32, x86-Linux, etc.) and improve the reusability of linker command files, the XDC Build Engine obtains the linker command file for an executable from the platform specified in the executable’s build script (unless overridden as described above).  By doing this, the XDC Build Engine allows a clean separation of platform-specific details from a portable (i.e., reusable) application code base.

Thus, an alternative to simply supplying a specific linker command file for a legacy application is to create a custom platform that supplies a linker command file that is appropriate for your application.  Why bother?  It is then possible to generalize and reuse the linker command file (using XDC template support) to support more than one executable; rather than copying a sophisticated linker command file and modifying it for each executable, a single linker command file template can be maintained.  This significantly reduces the maintenance cost when changes and bug fixes need to be made to the linker command file for all executables supported by the platform.

3.   XDC Build verses …

How does XDC simplify an embedded development build environment?

Traditional solutions to embedded cross development build invariably rely on the make build tool.  This is understandable because of make’s ability to efficiently coordinate the execution of disparate command-line based tools and its ability to accommodate virtually any physical organization of source code.  Over the years since make was originally created in 1976 [Feldman 1979], make has evolved a variety of sophisticated capabilities that have enabled it to remain one of the best build engines for embedded development.  Unfortunately, with sophistication comes the problem that it takes an expert to take full advantage of the capabilities of a modern form of make.

The XDC Build Engine generates GNU makefiles from a high-level specification of what is in an XDC package.  These generated makefiles are then used to control the build.  The generated makefiles take full advantage of make’s sophisticated capabilities without the developer needing to be a makefile expert; e.g., parallel and hierarchical builds, automatic dependency generation, side-by-side build for multiple targets, etc.

Because the build is entirely controlled by GNU make, it is very easy to integrate legacy makefiles into an XDC package.  It is also very easy to extend an XDC build using a standard proven build language (the language of GNU make).  Thus, the XDC Build Engine has all of the power of make while making it easy for developers, who are not makefile file experts, to take full advantage of the advanced features of make.

Why not CCS or Microsoft’s VisualStudio project based build?

Project based build environments work well for one-man projects but they don’t scale well for development programs that require consistent use of tools across multiple member of a team.  Worse, these tools typically tie the developer to a single host development environment and a single target and platform.  Portable embedded code requires support for side-by-side builds for multiple targets and platforms.

Why not GNU make, nmake, pmake, clearmake, …?

While make has evolved into a universal build tool with sophisticated features that can handle virtually any type of build environment, it takes an expert in the subtleties of makefile syntax and dependency semantics to create a makefile that is both correct and takes full advantage of make’s capabilities. 

Most developers do not have time to become makefile experts and produce the content that the makefiles build.  More often than not, there is one developer on each development team that designs and services a makefile-based environment for the entire team.

In addition to its complexity, make (in any of its forms) is incomplete: production makefiles invariably require a Unix-like shell to compose the commands that are required to build the various goals and automatically generate header file dependencies.

Despite these deficiencies make has flourished in the embedded development community and is almost universally used.  The make engine has proven to be very effective at coordinating the execution of tools from dozens of different vendors.  Moreover, it provides powerful extension mechanisms to handle virtually any build requirement. 

This is why XDC uses GNU make as a “backend” build engine and treats the makefile statements as a type of assembly language that is generated from a high-level specification (XDC build scripts).  Experts can still extend the build environment using the makefile syntax, and non-experts can easily generate production makefiles that support parallel builds, side-by-side builds for multiple target and platforms, hierarchical builds, and automatic dependency generation.

Why not Ant or scons?

The XDC Build Engine is philosophically very similar to build tools such as Ant and scons.  In both of these tools, make’s declarative syntax is replaced by a standard procedural programming language; Java in the case of Ant, and Python in the case of scons.  Rather than construct subtle dependency chains and use a variety of shell script languages to specify what to build and in what order, these tools take the position that it is much easier and safer to simply use a standard procedural language with proven structured programming mechanisms (functions, arrays, flexible numerical and string types, rich built-in support functions, etc.).

So why do we need a third alternative?  Ant and scons are both relatively new tools and have a much smaller installed base of users than GNU make.  Neither supports legacy makefiles.  Moreover, Ant is currently focused on the management of “native” builds; i.e., builds of programs and libraries for use on the same platform that runs the development environment.

The XDC build engine is designed from the ground up to support the embedded-cross-development environment.  In this environment, one often needs to simultaneously build the program for execution on both the development platform and one or more embedded systems using disparate tool chains from multiple vendors.  For example, one may build an audio codec library for execution on a Wintel development system and one or more TI DSP families (e.g., C5000, C6000).

Like XDC, scons abstracts the platform in such a way that it cleanly supports side-by-side builds for multiple platforms.  In addition, scons abstracts build tools so that, like make, it can support the build of any type of file from any other type of file.  This makes scons an appropriate alternative back-end engine for the XDC Build Engine which, unlike scons or Ant, takes advantage of the XDC package structure to simplify build scripts and unify deployment of target content.

4.   Configuring the Build Model

How do I tell the XDC Build Engine where the XYZ compiler is installed?

The Build Object Model is configurable and is configured using a script named config.bld.  This script is located by searching the following locations in order: the directory containing the current package’s package.bld script, the directories named in the package path, the XDC installation directory, and the etc sub-directory of the XDC installation directory.  In this script, which is executed prior to any package’s build script, it is possible to specify all targets and platforms used in a build.

For each target in the Build.targets array, you must set the rootDir configuration parameter.  This configuration parameter specifies the absolute path to the installation directory of the compiler tool-chain used by the target to build libraries and executables.  In the fragment below, the first line defines a variable that refers to a package of targets.  The second line sets the rootDir parameter of the C62 target in this package.

var C62 = xdc.useModule("ti.targets.C62");
C62.rootDir = "c:/ti/c6000/cgtools/";

Notice that file names always use ‘/’ in lieu of ‘\’ to delineate directory names.  Since the ‘/’ character can be used on Windows systems as well as *nix systems, this form of file name is portable.

How do I configure path names with embedded spaces?

You can’t.  Path names must not contain spaces.  If you must use a path name with embedded spaces and you are using a Windows development host, you can replace the path name containing embedded spaces with the equivalent 8.3 name; use the “dir /x” command from a DOS prompt to see the equivalent 8.3 name of any file or directory.

How do I control the set of targets used in a build?

There are several ways to control the set of targets used in a build.  Each package build script can explicitly name the targets that should be used when building the package.  For example, the following fragment builds the library “foo” for the ti.targets.C62_big_endian target.

Pkg.addLibrary("foo",
    xdc.om["ti.targets.C62_big_endian"]);

Packages of largely portable code typically build for all targets specified in the Build.targets array; this allows one to build for a new target by modifying config.bld rather than every package’s build script.  In this case, the package’s build script will follow the pattern shown below.

for (var i = 0; i < Build.targets.length; i++) {
        :
    Pkg.addLibrary(…, Build.targets[i]);
}

In this case, the Build.targets array must be initialized outside the package’s build script and this is typically done in the Build Object Model configuration script, config.bld.  For example, the fragment of config.bld shown below first loads the target module “ti.targets.C62” and then it adds it to the Build.targets array.

var C62 = xdc.useModule("ti.targets.C62");
Build.targets[Build.targets.length++] = C62;

If after running config.bld the Build.targets array is empty, the environment variable XDCTARGETS is used to populate this array.  The XDCTARGETS environment variable is interpreted as a space separated set of regular expressions that are used to select from all available targets.  For example, the following setting will initialize the Build.targets array to include all targets whose name begins with "ti.targets.C62".

export XDCTARGETS="ti\\.targets\\.C62.*"

For more information about using XDCTARGETS to control the set of targets, see the xdc Command Reference man page.

How do I configure a target?

Targets are represented by any module that implements the xdc.bld.ITarget interface.  This interface (or any extension of it) defines numerous “configuration parameters”.  These parameters allow one to adjust the behavior of the target in your development environment.  For example, you can configure the file extensions understood by a target, the installation directory of a target’s compiler and related build tools, the name of the default platform associated with a target, etc. 

For a list of all configuration parameters supported by all targets see the xdc.bld.ITarget interface definition file (xdcroot/packages/bld/ITarget.xdc).  For a complete list of a particular target’s configuration parameters you must review the target’s interface definition. 

Targets are configured in a config.bld file.  By configuring targets in this file, multiple packages can share a common configuration.  This makes it much easier to manage the development of multiple packages; one can change the set of targets in one place without having to modify every package’s build script.

How do I create a custom profile for a target?

A profile is a named collection of options for all of the tools in a target’s tools-chain.  Every target has a configuration parameter, named “profiles”, that is a “map” of profile names to tool-chain options.  This map is initialized in the config.bld script for each target.

var C62 = xdc.useModule("ti.targets.C62");
C62.profiles["debug"] = {
    compileOpts: {copts: "-g", defs: "-D_DEBUG=1"}
};

Profile names are then used in package build scripts (package.bld) to select appropriate modes of operation from the targets.

  Pkg.addLibrary("foo", xdc.om["ti.targets.C54"], {profile: "debug"});

How do I control the set of platforms used in a build?

As with the specification of targets, the set of platforms may be specified either in a package’s build script, package.bld, or in the Build Object Model’s configuration script, config.bld.  Package build scripts specify platforms using the platform’s name.  For example, the package.bld fragment below builds the executable “foo” for the ti.platforms.dsk5416 platform.

Pkg.addExecutable("foo", xdc.om["ti.targets.C54"],
    "ti.platforms.dsk5416");

Because platforms can, in general, execute code built using many different targets, one must specify both a target and a platform; e.g., an ARM platform can run executables built using ADS, GNU, or TI compiler tool-chains.  Yet not every combination of target and platform are compatible; e.g., a DSK6416 can run both big and little endian executables, but clearly can not run code built using a C55 target.  Since this relationship between platforms and targets can be complex and may change as new targets are created, it is desirable to specify legitimate combinations in one place that is shared by multiple package build scripts.

Each target has a platforms configuration parameter which allows one to specify a set of platforms that are compatible with the target; thus, it is possible to build executables for a new platform by modifying config.bld rather than every package’s build script.  Suppose, for example, that each package’s build script follows the pattern shown below.

for (var i = 0; i < Build.targets.length; i++) {
  var targ = Build.targets[i];
    for (var j = 0; j < targ.platforms.length; j++) {
        :
      Pkg.addExecutable(…, targ, targ.platforms[j]);
    }
}

In this case, the set of platforms to use is specified in config.bld by configuring each target’s platforms array.  Doing this ensures that as new targets and platforms become available, individual package build scripts (for portable executables) do not need to change.

5.   Building Packages

How do I debug a build script?

Build scripts are used to generate makefiles.   These scripts are written in the JavaScript (ECMA-262R3) language with a few extensions to facilitate debugging and partitioning of scripts into multiple files.  At any point in a script, one can embed a call to the “print” function.  This function converts all arguments into a string and outputs the result to stdout.

In addition to creating a makefile from the build script, the XDC Build Engine generates an XML file (package/package.bld.xml) that describes the contents of the package.  This file can be reviewed, using any XML reader or modern web browsers, to ensure that the build script properly expresses the entire collection of files to be built.

Finally, since build scripts are run via the tconf tool and this tool has a built-in graphical debugger, it is also possible to single-step debug build scripts.  Recall that the xdc command is simply a “wrapper” for GNU make, so we need a way to tell make to pass the “-g” option to tconf whenever tconf is used to run a build script.  This can be done by either setting the environment variable MK_BLDOPTS to “-g” or by directly setting this variable on the xdc command line.

xdc MK_BLDOPTS=-g …

How do I build multiple packages at once?

The xdc command can be used to build multiple packages.  The ‘-P’ or ‘-PR’ options are used to name the packages to be built.  The ‘-P’ option is used when you want to explicitly name each package to be built.

xdc –P examples/basic/hello examples/basic/misc

The ‘-PR’ option is used when you have one or more directories that contain packages (possibly at different levels) and you want to build all packages in these directories.

xdc –PR examples

The xdc command finds packages by locating a directory that contains both a package.xdc and a package.bld file.

How do I build for just one target?

During development of a portable package (i.e., a package that can support multiple targets and platforms), it is often desirable to build for just one target during debugging.  There are several ways to control the set of targets used in a build:

1.      define the set of targets in the build startup script (config.bld);

2.      use the XDCTARGETS environment variable; or

3.      use target specific “abstract” goals (e.g., “all,62”, “.libraries,55”, etc.).

The first two options require modification of the build startup script; either to explicitly initialize the Build.targets array or to remove these settings so that the xdc command will look to the XDCTARGETS environment variable.  If multiple developers share the same build startup script, this can be problematic.  Worse, changes to the build startup script trigger a rebuild of virtually all files.

The third option allows the developer to limit a build based on a particular target without requiring a regeneration of makefiles.  For example, the following command builds all files related to the target whose suffix is “62”.

xdc all,62

For a complete list of “abstract” goals supported by the generated makefile see the xdc Command Reference.

How are inter-package dependencies handled?

When multiple packages are built at once, each package is built in several “passes”: first makefiles in each package are built, followed by interface files, libraries, dlls, and executables.  By building each package in several passes, it is possible to build packages that depend on one another in any order.

Are circular package dependencies allowed?

No.  Circular dependencies between modules may exist within a package but never between packages.

Why aren’t complete build commands displayed?

Displaying the complete commands makes it difficult to spot the command being executed and identify what file is being compiled, linked, assembled, etc.  The full command used to compile a source file can be hundreds of characters long whereas the name of the file being compiled is often no more than 10 characters long.  So, the default output of the build is a single line “abbreviation” of each significant operation and the name of the file being processed.

However, there are times when it is important to review, in detail, every option and every command used to generate a build goal.  To see the complete commands as they are being executed, set the XDCOPTIONS environment variable to ‘v’ (for verbose).

set XDCOPTIONS=v

If you only want to see the commands for a particular run of the xdc command, you may set XDCOPTIONS on the command line.

xdc XDCOPTIONS=v

Why are my sources being re-built every time I change package.bld?

In general, any change in package.bld could change the compiler options that would require re-compilation.  The XDC Build Engine currently generates a separate target-specific makefile for every library and executable.  Each object file associated with a library (or executable) has a dependency on the library’s (or executable’s) generated makefile which triggers a re-compilation for any change to this makefile.  While this ensures that any change in compiler options will trigger a recompile, it may also trigger unnecessary compilations.

Why did my package rebuild even though none of its sources changed?

There are several environment variables that may affect the contents of the generated makefile; e.g., XDCPATH, XDCTARGETS, XDCBUILDCFG, etc.  If any of these variables changes between successive builds, this will trigger a re-generation of the makefile that, in turn, will trigger recompilation of the package’s sources.  Note that changing XDCOPTIONS does not trigger makefile re-generation; XDCOPTIONS only controls what is displayed during the build not the contents of any build goal.

If your package uses modules or interfaces from another package and that package’s interfaces change, this can also trigger a re-build of your package.

Why doesn’t the xdc command catch misspellings of XDCTARGETS?

The xdc command is simply a wrapper program for running GNU make with a standard set of options.  In order to allow the user to fully access the power of GNU make, the xdc command passes all unrecognized options directly to GNU make.  Thus, a misspelling of XDCTARGETS cannot be distinguished from a new variable definition set by the user and used in a custom makefile included by a package’s makefile.

Why do some packages have empty build scripts?

Some packages are delivered (and used) in a binary form without sources whereas others are delivered in a form that allows them to be rebuilt using newer tools; the xdc tools use the existence of a build script to identify “buildable” packages.    Packages with empty build scripts are packages that only contain interfaces or meta-modules that are already declared in the package’s specification file, package.xdc.  Without a package.bld script, such packages can not be rebuilt using a newer version of the xdc tools.

Does xdc support parallel builds?

Yes.  Use the GNU make ‘--jobs’ option to control the amount of parallelism to use during each build.

xdc -–jobs=4

Although the generated makefiles support parallel builds, not all tool-chains do.  For example, some TI compilers that run on Windows do not generate unique temporary files when compiling the same source file.  Thus, it is not always possible to use parallel build on a Windows PC.

How do I create a “debug” and “release” variant of a library or executable?

Each target supports multiple named “profiles” which correspond to a set of compiler, assembler, and linker options.  One can configure each target with a “debug” and “release” profile in config.bld (see the section “Configuring the Build Model”).  Package build scripts can then use these profiles to build library, dlls, and programs.

Pkg.addLibrary("hello_release", ti.targets.C54, {profile: "release"});

Pkg.addLibrary("hello_debug", ti.targets.C54, {profile: "debug"});

Notice that you must create a different library name for each different variant and you must also create a custom getLibs function for your package.  For the example above, you must add the following definition to the file named package.xs.

function getLibs(prog)
{
    /* use this package’s profile setting to select directory */
    var name = this.profile == "debug" ? "hello_debug" : "hello_release";

    /* return the library name: name.a<arch> */
    return (name + ".a" + prog.build.target.suffix);
};

For a complete description of this function see the xdc.IPackage interface specification file located in xdcroot/packages/xdc.  For more information about controlling libraries, see the “Using Other Packages” section of this FAQ.

How do I add a compiler option to the “debug” (or “release”) profile of all targets?

Each “profile” defined by a target may be modified in the config.bld script used to configure the targets (see the section “Configuring the Build Model”).  Suppose, for example that you want to add “‑D_CHECKED_=1” to all debug profiles for all targets configured in the Build.targets array.  The following script fragment, when added after initialization of the Build.targets array, will do the trick.

for (var i = 0; i < Build.targets.length; i++) {
   var profile = Build.targets[i].profiles["debug"];
   profile.compileOpts.defs += " -D_CHECKED_ ";
}

Since target profiles often already contain some standard options, the script above simply adds to what ever is already defined by the target.  A similar script fragment can be used to remove or modify existing options.

How do I add target-specific statements to an otherwise portable C/C++ source file?

Every source file is compiled with a set of predefined macros that enable you to add target-specific statements to otherwise portable C/C++ sources. 

These predefined macros are designed to maximize the portability of the resulting sources by avoiding (as much as possible) target-specific names.  For example, you may have a portion of a source file that requires a C6000 processor but can work equally well on a C62, C64, C64P, C67, etc.  In this case, one can use the “isaCompatible” macros.

#ifdef xdc_target__isaCompatible_62

    : /* code that works on any C6000 device */

#endif

#ifdef xdc_target__isaCompatible_64

    : /* code that works on C64 or C64P devices */

#endif

The predefined macros are described in the xdc.bld.ITarget interface; see the documentation of the xdc.bld.ITarget.CompileOpts structure.  Since each target defines a set of compatible ISA’s (Instruction Set Architectures), you must also refer to each target’s reference documentation to obtain the complete list of predefined macros.

How do I create “debug” and “release” variants in a directory named “debug” and “release”?

The name parameter passed to addLibrary may include directory prefixes.  So, you can use a different directory prefix for the libraries:

Pkg.addLibrary("release/lib", ...);

Pkg.addLibrary("debug/lib", ...);

In this case, the release archive is release/lib.a62 and the debug archive is debug/lib.a62.

How do I pass command line arguments to a build script?

It is possible to pass command line arguments to a build script using the XDCARGS environment variable (or command line setting).  The value of XDCARGS is passed to the build script in the form of arguments accessible via the JavaScript arguments[] array.  For example, the following command will initialize arguments[0] to the string “regress”.

xdc XDCARGS=regress

The build script can then conditionally execute based on this value.

if (arguments[0] == "regress") {
    :
}

How do I specify different compiler options for different files in a package?

It is possible to specify file-specific compiler options by setting the appropriate xdc.bld.Object.Attrs parameter of each xdc.bld.Object added in a build script.  The following example causes the “-D_DEBUGLEVEL=3” compiler option to be used when compiling the source files “foo.c” and “bar.c”.

var lib = Pkg.addLibrary("mylib", …);
lib.addObjects(["foo.c", "bar.c"], {
    copts: "-D_DEBUGLEVEL=3"
});

Many of the objects in the Build Object Model are “containers” of other objects; e.g., the xdc.bld.PackageContents object contains xdc.bld.Library and xdc.bld.Executable instances, and xdc.bld.Library and xdc.bld.Executable instances contain xdc.bld.Object instances.  This containment relationship is used to facilitate the specification of file-specific options; if compiler options are set for a container, all objects contained by this object will “inherit” the same options unless explicitly overridden.

For example, to use the “-D_DEBUGLEVEL=0” compiler option for all files in a package, it is sufficient to set the xdc.bld.PackageContents.Attrs.copts parameter of the Pkg object; recall that Pkg is an alias for the xdc.bld.PackageContents singleton.

Pkg.attrs.copts = "-D_DEBUGLEVEL=0";

To override this setting for all files compiled and archived in a particular library in the package, set the xdc.bld.Library.Attrs.copts parameter of the library instance object created via the xdc.bld.PackageContents.addLibrary method.

var lib = Pkg.addLibrary("mylib", targ, {
    copts: "-D_DEBUGLEVEL=1"
});

To override this setting for an individual object file in this library, set the xdc.bld.Object.Attrs.copts parameter of the xdc.bld.Object instance created via the xdc.bld.Library.addObjects method.

lib.addObjects(["foo"], {
    copts: "-D_DEBUGLEVEL=2"
});

Why are source file extensions sometimes omitted in build scripts?

If a source file extension is omitted, the XDC Build Engine determines an appropriate source file by using the first file that matches the name specified in the build script followed by an extension named in target’s “extensions” configuration parameter.

This mechanism makes it possible to add an object to a library and have a different assembly language source files for each target.  For example, suppose you have three assembly language implementations of sqrt(); sqrt.s62, sqrt.s54, and sqrt.s55.  The following build script assembles the appropriate file for the each target-specific library.

for (…) {
    var lib = Pkg.addLibrary("math", targ);
    lib.addObjects(["sqrt"]);
}

This mechanism also implies that no change to the build script is necessary when converting an implementation in one language to another; e.g., converting C to assembly language.

Why are there so many different file extensions after a package has been built?

The XDC Build Engine enables “side-by-side” compilation of portable sources using a variety of compilers.  Since the output of each compile and link must be stored in a different file, the build engine simply uses a different extension for each target (the extension is specified by the target’s implementation of the xdc.bld.ITarget interface).

By default, the XDC Build Engine uses the following extensions:

.a<suffix>         archives of objects built using the target whose suffix is <suffix>

.o<suffix>         object files built using the target whose suffix is <suffix>

.x<suffix>         executables linked using the target whose suffix is <suffix>

.s<suffix>         assembly source that can be assembled using the target whose suffix is <suffix>

If, like the DSP/BIOS team, you build a portable source file with 15 different targets you will have over 45 (3 * 15) different extension types in your package.

Why not use a common extension for object files, executables, and archives?

Using a single extension makes it easier to make mistakes; e.g., inadvertently use a big endian object file in a little endian build.  It also makes it difficult to easily identify the type of a file if it gets moved; e.g., when copying executables from one place to another.

Perhaps the most important reason, however, is that doing this would make it impossible to associate other tools with these types of files.  Suppose, for example, that you have a graphical COFF browser tool that only works on TI object files.  If all objects had the same extension, say “.obj”, it would be impossible to associate a different tool for the non-TI object files; double clicking on an “.obj” can only run one tool.

How do I use tools that require specific extensions?

In most cases, it is possible to configure the XDC Build Engine to generate/recognize files with extensions you specify.  For example, it is possible to configure each target to generate executables with an extension you specify; for a complete description see xdc.bld.ITarget.execExt.  In addition it is possible, on a per library basis, to specify the extension of the library; for a complete description see xdc.bld.Library.Attrs.suffix.

How do I create a package that supports a single target or platform?

In this case the package.bld script simply names the target and or platforms directly.  For example, the following build script build the hello executable for just the DSK5416.

var C54 = xdc.useModule("ti.targets.C54");
Pkg.addLibrary("hello", C54);
Pkg.addExecutable("hello", C54, "ti.platforms.dsk5416");

6.   Using Other Packages

What is the “package path” and how is related to XDCPATH?

The package path names one or more repositories (i.e., directories that contain packages).  When building or using a package, the XDC tools will search for a package’s files by looking for the package’s directory in the repositories named in the package path. Thus, the package path does not name package directories; it names directories containing package directories.

The XDC package path is constructed from the environment variable XDCPATH, the installation directory of the XDC tools, and the current package being built.  In particular, the package path is defined as follows:

package path ≡ $XDCPATH;xdcroot/packages;cur-package-repository

where $XDCPATH is the value of the environment variable XDCPATH, xdcroot is the installation directory of the XDC tools, and cur-package-repository is the repository containing the package currently being built.  The XDCPATH environment variable allows one to add package repositories that are distinct from the repository containing the current package and that are not part of the XDC distribution.

Suppose, for example, that all packages from third parties are installed in /usr/local/packages and you are also developing several packages of you own.  The following setting of the XDCPATH will allow packages to use any package in /usr/local/packages or your own packages.

export XDCPATH="/usr/local/packages;/usr/home/me"

Repositories are always separated by the ‘;’ character; this is also true for *nix hosts.

Note that it is usually a mistake to put a relative path in the XDCPATH environment variable.  Relative paths in XDCPATH reference directories relative to the package being built rather than the directory where the xdc command was invoked.  Thus, a relative path will refer to a different repository for each package being built.

Do I have to use the XDCPATH environment variable to control the package path?

Environment variables, such as XDCPATH, that can impact a build’s results are problematic in team development environments.  It is very important that code built by one developer can be re-built from scratch by another developer with the same results.  If a build’s results are a function of environment variables, it can be difficult to ensure consistent builds among members of the development team.

A package’s build script may explicitly override XDCPATH.  In this case, and the package’s build results will not be dependent on the developer’s setting of XDCPATH in the environment.  The following build script fragment illustrates how this can be achieved.

Pkg.packagePath = "/usr/local/packages";

It is also possible to define the package path in terms of a different environment variable.  Suppose, for example, that all developers use a single environment variable, say TOOLS, to locate a global repository of development tools.  One can define a package’s package path in terms of this environment variable as follows.

Pkg.packagePath = "$(TOOLS)/packages";

The expression $(TOOLS) is standard make syntax for the value of the macro TOOLS; make converts all environment variables into macro definitions prior to evaluating other expressions in a makefile.

Can multiple versions of the same package appear along the package path?

Yes.   Although the package directory name must match the package’s name, the package path can name multiple repositories that can contain a package directory with the same name.  Currently, no check is made to ensure the all package directories named in the package path are unique.

How do I refer to headers in another package?

The package’s directory name prefix must be used in the #include statement.  Suppose, for example that you need to use the Tsk module in a package named “ti.bios”.  To use the Tsk module’s methods from a C/C++ file you must add the following statement to the top of you source:

#include <ti/bios/Tsk.h>

How do I refer to headers in another package without naming the package in my source?

It possible to avoid using a package’s name when including one of its header files by declaring that the package is “used” in your package.bld script.

Pkg.uses = ["ti/bios", "ti/bridge"];

The package path will be searched for a directory that matches the path suffixes named in the “uses” array and these directories are added to the C/C++ compiler's include paths.  Thus, to use the Tsk module’s methods from a C/C++ file it is sufficient to have the following statement to the top of you source:

#include <Tsk.h>

How do I refer to configuration scripts in another package?

As with C/C++ header files, one must use the package’s directory name prefix when referring to script in another package.  Suppose, for example that you need to use the tsktest.cfg configuration script from the “ti.bios.testsuite” package.  To use this script, you must reference the script as “ti/bios/testsuite/tsktest.cfg” in the package.bld script:

Pkg.addExecutable("test", targ, targ.platform, {
    cfgScript: "ti/bios/testsuite/tsktest.cfg"
}

How do I link with libraries in another package?

Libraries are automatically included on the link line for any package that is used by a program.  All packages loaded during configuration of a program are queried after the configuration script completes for a list of libraries that should be added to the link line.

The line below, if added to a program configuration script, will cause the resulting executable to be linked with the libraries from the “ti.mypackage” package.

xdc.loadPackage("ti.mypackage");

If you are creating a package that requires another package, you should declare this in your package’s package.xdc file.  For example, if the package “app” requires the package “ti.bios”, the app package’s package.xdc file declares this dependency as follows:

requires ti.bios;
package app { …}

How do I control which libraries are used when linking an executable?

The set of libraries used to link an executable is determined after the program’s configuration script completes and before the executable’s “linker command” file is generated; this file contains linker options used to link with this set of libraries and produce the executable.  The XDC Build Engine builds this list of libraries by visiting every package referenced (directly or indirectly) by the program’s configuration script and calling each package’s getLibs function.  For a complete description of this function see the xdc.IPackage interface specification file located in xdcroot/packages/xdc.

You can select, on a per package basis, which profile of each package you’d like to use within an executable.  For example, the following configuration script fragment selects the "debug" profile of just the package containing the ti.sysbios.knl.Task module.

var Task = xdc.useModule("ti.sysbios.knl.Task");
Task.$package.profile = "debug";

Alternatively, you can get the package object via the xdc.loadPackage() method in order to set the package's profile property.

var bios = xdc.loadPackage("ti.sysbios.knl");
bios.profile = "debug";

How do I link with libraries that are not in an XDC package?

The cleanest way is to create a package that encapsulates the libraries that you need.  The package you create can, as part of its build, copy the libraries from a known installation into the package’s directory.  The package can then supply a package.xs file that implements the getLibs() method.  In this method, you simply return the appropriate library name (relative to the package’s directory) for the executable that is being built.

Alternatively, you can simply add the appropriate link options to the executable being built.  This has the advantage of being quick and easy.  However, it requires that you know the appropriate linker options for every target you are using.  In addition, every package that wants to use this library will have to incorporate this same target specific information in their build scripts.

7.   Creating Executables and Running Tests

How do I use xdc to build a “legacy” executable?

The legacy.hello example illustrates how to build an executable that does not use any configurable modules.  This is indicated in the package’s build script (package.bld) by setting the executable’s cfgScript attribute to null.

Pkg.addExecutable("hello", targ, plat, {

    cfgScript: null     /* no configuration script */

});

The only difference between executables built using traditional tools and the XDC tools is that the XDC tools require all executables to have a configuration script.  Configuration scripts name modules used by the executable and specify a particular configuration of these modules.  Upon completion of a configuration script, several files are generated from the “model” of the application created by the script.  These files must then be compiled and linked into the executable.

Note that setting the executable’s cfgScript attribute to null is not the same as specifying an empty configuration script.  Setting the executable’s cfgScript attribute to null is equivalent to specifying the following configuration script:

Program.main = null;

Program.system = null;

For more information about these settings see the xdc.cfg.Program module’s reference documentation.

Why are some program configuration scripts empty?

An empty configuration script simply means you are happy with the default configuration settings for the implicitly loaded packages:

  1. the package containing the configuration script;
  2. the Run-Time Support package named by the target (see xdc.bld.ITarget.rts); and
  3. the xdc.cfg package.

 

We could assume that if the script does not exist then you meant to specify the empty configuration script.  However, a simple typo in the name of the configuration script would result in no warning message from the tools and an errant executable being created.  Thus, a non-existent configuration script is an error.

How do I pass arguments from a build script to a program configuration script?

See the cfgArgs description in the xdc.bld.Executable.Attrs reference manual.

How do I supply my own linker command file for a particular executable?

Since the platform defines the memory map for any executable that runs on the platform, the default linker command file for an executable is obtained from its platform; see xdc.Platform.getLinkCmd (defined in xdc/Platform.xdc).  However, some executables require sophisticated application-specific control over placement; e.g., to implement overlays of algorithms from separate packages.

In this case, the developer’s program configuration script may override the default by setting the linkTemplate configuration parameter of the Program object; for more information see xdc.Program.linkTemplate (defined in xdc/Program.xdc).

Program.linkTemplate = "myPackage/overlay.xdt";

The package legacy.lcmd, located in the examples sub-directory of the CDK, illustrates this technique.

What is a test and how is it different from an executable?

A test is an executable together with command line arguments to be passed to the executable.

How do I create tests in an XDC package?

Each time an executable is created in the package’s build script, a test is implicitly added to the package; this test simply runs the executable without any arguments.  Thus, it is impossible to add an executable without at least one test.

It is possible to create additional tests using the xdc.bld.Executable.addTest method.  The following build script fragment creates a second test for the executable foo.

foo = Pkg.addExecutable("foo", targ, plat);
foo.addTest({args: 'these are arguments to foo'});.

How do I run tests defined in an XDC package?

The following command runs all tests in a package.

xdc test

The following command runs all tests in all packages located in or below ".".

xdc test –PR .

For more information see the xdc reference man page.

I have hundreds of tests and when one fails the remaining are not run, how do I run all tests even though one (or more) fails?

Recall that the xdc command is a simple front-end to GNU make.   Thus, it is possible to use standard GNU make options to control the how xdc behaves when a command fails.  In this case, it is possible to use the GNU make option ‘-k’ to tell xdc to keep going.

The following command runs all tests in a package even if one or more tests fail.

xdc –k test

For more information see the xdc reference man page.

How do I run only those tests that have an executable built with a particular target?

 The following command runs all tests associated with the target targ.

xdc test,targ

How do I control the order that tests are run?

The default execution order of tests is determined by the order that they are created in the build script.  You can affect this order by adding explicit dependencies in a custom makefile.  See the section “Extending the Build Model”.

8.   Deploying Packages

How do I deploy an XDC package?

The XDC Build Object Model has a notion of a release.  Each release corresponds to a tar file containing all files appropriate for the release.  Every package has at least one release (that is implicitly defined) that includes all libraries and interfaces required to use the package.

The following command builds the default release for a package.

xdc release

Can I create multiple releases for an XDC package?

Yes.  You can create a new release object via xdc.bld.PackageContents.addRelease().  The following command builds all releases of a package.

xdc release

Alternatively, you can build individual releases by pre-pending “release,” to the name of the release and pass this as a build goal to the xdc command.  For example, if a release is named “demo”, the following command will build just this release.

xdc release,demo

How do I build for multiple targets but separately deploy for each target?

Each library and executable may be part of multiple releases.  In the package’s build script, create a new release object via xdc.bld.PackageContents.addRelease() and add this object to the releases array associated with each library or executable built using the target.

var rel6x = Pkg.addRelease("6x");
    :
if (targ.isa == "62" | targ.isa == "64") {
    Pkg.addLibrary("mylib", targ, {
        releases: [rel6x]});
}

If the name of the release you created is “6x”, just this release can then be built using the following command.

xdc release,6x

How do I add sources to a release?

Set the attrs.exportSrc parameter of Pkg to true (it defaults to false).  If this parameter is set to true, all sources used to build executables and libraries added to the package will be added to the release.  Alternatively, it is possible to add all files that are not known to be generated to a package by setting the attrs.exportAll parameter of Pkg to true.  For more information see xdc.bld.PackageContents.Attrs section of the Model Reference Manual.

How do I add executables to a release?

Set the attrs.exportExe parameter of Pkg or an xdc.bld.Executable instance to true (it defaults to false).  If this parameter is set to true on the Pkg object, all executables added to the package will be added to the release.  If this parameter is set on the individual executable, then only that executable will be affected.

Pkg.attrs.exportExe = true;

Instead of (or in addition to) exporting executables you may prefer to export the configuration script that is used to build the executable.  Using the attrs.exportCfg parameter in exactly the same manner as attrs.exportExe above, program configuration scripts can be added to the release.

Pkg.addExecutable("go", targ, plat, {
    exportCfg: true});

How do I add arbitrary files to a release?

Set the Pkg.otherFiles parameter to an array of file names.  Every file named in this array will be included in the package’s release.

Pkg.otherFiles = ["README", "README.html"];

How do I add version information to executables?

Since every module used by an executable has an opportunity to generate arbitrary C data structures that are statically linked into the executable, it is possible to create a module that simply defines a global string variable that is set to the version of any selected package.  Recall that the C data structures are generated at the time the executable configured (i.e., just prior to linking) rather than at the time the module is created; so, the value of the version string can reflect the current version of the package containing the executable, for example.

Once such a module is created, say tisb.release.filters.Ident, you can embed version information into any executable by simply specifying that this module is used in the executable’s configuration script.  In this example, the following line is sufficient:

xdc.useModule('tisb.release.filters.Ident');

How do I add version information to libraries?

Every library of every package contains an object file, the compiled form of package/package.c, which contains information about the package containing the library.  The xdc.bld.PackageContents module allows one to specify a template that is expanded into a file that is included by package/package.c; see xdc.bld.PackageContents.libTemplate.  When the following line is added to a package’s package.bld script, the template “myVers.xdt” is expanded into a file (package/package.bld.c) that is included by package/package.c:

Pkg.libTemplate = "./myVers.xdt";

Using the predefined macros in package/package.c it is possible to create uniquely named variables that can be accessed at runtime.  For example, the following template creates several variables that reflect the current package’s version.

#define _concat(p,n) p ## n

#define myPrefixedVar(p,n) _concat(p,n)

 

/* create a version string that can be used at runtime */

char * myPrefixedVar(__xdc_PKGPREFIX, _versionString) = "`Pkg.version`";

 

/* create numeric version array that can be used at runtime */

float myPrefixedVar(__xdc_PKGPREFIX, _versionNumArray)[] = {`Pkg.version`};

 

/* create a "what" string that will be reported by vers utility */

char * myPrefixedVar(__xdc_PKGPREFIX, _whatString) = "@(#) " "`Pkg.name`" \

                             "; version " "`Pkg.version`" \

                             "; " __DATE__ " " __TIME__;

If your package’s name is foo.bar, the template above will create the following three variables:

1.      foo_bar__versionString, a string with the package’s declared version (e.g. “1, 0, 0”)

2.      foo_bar__versionNumArray, an array of float’s containing the numeric value of each component of the version string above, and

3.      foo_bar__whatString, a string containing package information that can be retrieved using the vers utility.

These variable definitions are included by the package/package.c file which is compiled and archived into *every*  library that is part of the foo.bar package.  Adding the package prefix ensures that these variables do not conflict with similar variables in other packages.

How do I add version information to arbitrary text files in a package?

Every file in any release of a package may be optionally filtered using a “release script”.  Release scripts run in a special “Release” model which contains the object xdc.bld.Manifest object.  This object lists every file that is part of the release.  Release scripts may specify a unique filter to be applied to each file before the file is added into the release archive; see xdc.bld.Manifest for more details.

 

It is possible to specify a unique release script for each release produced by a package (see xdc.bld.Release.Attrs) or a single release script that is used for all releases (see xdc.bld.PackageContents.Attrs).

9.   Extending the Build Model

How do I extend the build to include my own rules and goals?

Since the build engine behind the Build Model is standard GNU make (version 3.80), it is possible to extend the build environment using virtually any facility provided by GNU make.  The package build script below specifies that the generated makefiles should include the custom make file named "custom.mak". 

Pkg.makePrologue = "include custom.mak";

How do I add new targets?

Create a package with a module that implements the xdc.bld.ITarget interface and put this package in the package path.  Adding a new target should be a last resort, however.  The automatic target compatibility checks only validate consistency among like-named targets; thus, if an executable is built using libraries build using targets of different names, no validation will occur between these targets.

Although the xdc.bld.ITarget interface is relatively straightforward, it is usually better to inherit from an existing implementation than to create one from scratch.  The example local.targets shows how one can easily create new targets by inheriting target implementations from the ti.targets package.

After adding a new package of targets it is usually necessary to also add a runtime support package that contains pre-built libraries of the sources found in the xdc.rts package.  This runtime support package provides low-level services that virtually every program can benefit from; e.g., primitive printf support, clock services, and configurable program exit() and abort() methods.  In addition, it provides a default boot startup required by the C runtime.

How do I build packages of targets?

Packages of targets are built the same way as any other package.  However, there is one subtlety: the build engine obviously can’t use targets contained in this package while building this package (because they have not yet been processed by the RTSC IDL compiler.)  Thus, it is necessary to tell the Build Engine to not use this package when building this package.  This is accomplished by adding the file config.bld to the top of the package which contains the following line:

Build.useTargets = null;   /* make the set of targets empty */

Packages of targets often have empty build scripts.  To understand why, see the FAQ “Why are some package build scripts empty?”.

Who uses the target runtime support libraries?

Any program that references the services provided by the xdc.rts package’s interfaces that also does not have a startup “boot file”.  Thus, legacy programs which have their own boot startup file do *not* need to use the runtime support library.  A legacy program is any program that was previously built without RTSC. 

Programs that rely exclusively on DSP/BIOS for basic services do *not* require an xdc.rts runtime support library; DSP/BIOS supplies a boot startup file.

How do I add a new target runtime support package?

The example package local.targets.rts470 shows how one can easily create a runtime support package leveraging the sources contained in the xdc.rts package.  This package must be delivered as a source package to allow end-users to rebuild the libraries in the event that a new, possibly incompatible, compiler becomes available before a new pre-built version of this package becomes available.

 There are two unusual aspects to this package: even though this is a source package the sources are not contained in this package, and two libraries are created for each target.   Sources for the library are always maintained in the xdc.rts package; this allows the xdc.rts package to be independent of new or updated runtime support packages.  The boot startup file is placed in a separate library from the other sources to allow programs to provide their own boot startup and still take advantage of the xdc.rts modules.

Both aspects are handled in the package’s build script, package.bld.  To compile sources contained in the xdc.rts package, package.bld loads the xdc.rts package and allows it to add content to the generated makefile:

var RtsBuild = xdc.loadCapsule('xdc/rts/build.xs');

Pkg.makePrologue = RtsBuild.makePrologue;

 

Building two libraries per target is simply matter of calling addLibrary() twice per target.

How do I add a new platform?

Create a package that contains a module named Platform that implements the xdc.IPlatform interface and put it in the package path.

10.         Troubleshooting

The xdc command doesn’t work from a Windows command shell, why?

The xdc command works from any command shell on Windows NT, 2000, or XP.  However, there are variations between shells that can cause the grief.

Unlike *nix shells, Windows shells do not strip quote characters (i.e., ‘’’ or ‘”’) when setting environment variables from the command line.  Thus, one should not use quotes when defining environment variables, even when setting values with embedded spaces, unless you what the quotes to be part of the definition. 

Thus, when setting the XDCTARGETS environment variable from a Windows shell it is important to not use quotes.

: build for targets C62 or C54 only
set XDCTARGETS=C62 C54

: wrong!  this specifies targets ‘"C62’ and ‘C54"’!
set XDCTARGETS="C62 C54"

My build failed, how do I view the commands use to build xyz?

The xdc command is a simple command line tool that executes the GNU make program with several required options.  You can view the GNU make invocation using the ‘-n’ option to xdc.

xdc –n hello.x62

If you want to see the commands used to build hello.x62 (without running the commands), use the GNU make option “--just-print”.

xdc –-just-print hello.x62

Finally, the makefiles generated by the XDC Build Engine display varying amounts of information based on the setting of the XDCOPTIONS environment variable.  You can view all commands executed during a build (as they are being executed) by setting XDCOPTIONS to ‘v’.

xdc XDCOPTIONS=v hello.x62

My link failed, where is the linker command file used to link my executable?

After a program’s configuration script completes, XDC generates a linker command file that is used to produce the executable.  The generated linker file is located in the package’s “package/cfg” sub-directory.  The name of the linker command file is

prog_xtarg.xdl

where, prog is the name of the program and targ is the target’s suffix.  For example, the linker command file for the “hello” program built using the ti.targets.C62 target is ./package/cfg/hello_x62.xdl.

This linker command file is generated from a “template” obtained from the platform package by calling the platform’s getLinkCmd() method.  This method is implemented in the package.xs file located in the platform’s package.

I’m using the TI linker, where is the map file for my executable?

Like the linker command file, the map file is in the package’s “package/cfg” sub-directory.  The name of the linker command file is

prog.xtarg.map

where, prog is the name of the program and targ is the target’s suffix.  For example, the linker command file for the “hello” program built using the ti.targets.C62 target is ./package/cfg/hello.x62.map.

Why do I get an arrayOutOfBounds error when adding to an array parameter?

All host objects in the Build Object Model are type safe and, like Java, it is an error to access an item outside an array’s pre-defined bounds.  Thus, in order to add an item to an array it is necessary to increase the array’s length before defining the new value.  The following statement increases the array length by one and defines the new array item:

Build.targets[Build.targets.length++] = newTarget;

This statement increases the length of the array by adding on to the array’s length.  The previous length is then used as the index to the newly created “slot” which is defined to be newTarget.

Executables that load my package are not linking with the right libraries, why?

Recall that each package controls, via the xdc.IPackage.getLibs function, which of the package’s libraries to link with for each executable produced.  Make sure that your package’s getLibs function is being called and that it is returning the right values.

If you didn’t define a getLibs function in your package’s package.xs file, make sure the default implementation of this function is appropriate for your package; the default function is defined in xdcroot/packages/xdc/package.xs.

I think there’s a compiler bug, how can I get an assembly listing of my C code?

All targets support the ability to retain the assembly source used by the compiler to produce the corresponding object file.  The makefiles generated by the XDC Build Engine contain a goal for each object file goal that allows one to generate the assembly source representation of the object file; if the object file goal is name.otrg, where name is the object file name minus the extension and trg is the target’s suffix, the assembly file source goal is name.strg.

Suppose, for example, that hello/Hello.o62e is the name of an object file produced as part of building a package.  The following command will generate the assembly language listing corresponding to this object file.

xdc hello/Hello.s62e

My executable’s generated linker command file names a non-existent library, why?

All libraries named in the linker command file are the result of calls to package getLibs functions.  If there is a library named in an executable’s linker command file that does not exist, some package has generated an invalid library name.

Note that if a package does not define a getLibs function, a default implementation is used.  The default implementation returns null if your package does not create any libraries; otherwise, it returns name/name.atrg, where name is you package’s name and trg is the target’s suffix.

Thus, you must define a getLibs() function if and only if you create a library that does not have the same name as your package.

I’ve added a library to my package and now my executables fail to link, why?

You probably need to write a customized getLibs function for your package.  Recall, you must define a getLibs function if and only if you create a library that does not have the same name as your package.

I’ve added a relative path to XDCPATH and now my multi-package builds fail, why?

Relative paths in the package path (XDCPATH) are evaluated relative to the package currently being built not the current working directory of the xdc command.  So, unless the relative path resolves to the same directory for all packages in a multi-package build, some packages will see different actual package directories along the package path.

In general, paths named in XDCPATH should be absolute paths.

xdc seems to require forward slashes in file names, why?

Forward slashes (/) are the only portable means for separating directory names and file names.  UNIX systems use forward slashes and Win32 systems can use either forward or back slashes (\).

In general, even on Windows systems, all paths should use forward slashes to delimit directory and file names.
