Installation

System Requirements

The XDC tools are supported on the following development hosts:

Windows NT 4, 2000, and XP

Red Hat Enterprise Linux WS release 3 or above

Solaris 2.8 or above

If you are using one of the UNIX development hosts, you must also have a Korn Shell compatible shell installed as /usr/bin/ksh.

Unpacking the Distribution

The XDC tools are distributed as a single gzip’d tar file for each of the supported development hosts; xdc-win32.tar.gz for Windows, xdc-linux.tar.gz for Linux, and xdc-solaris.tar.gz for Solaris. 

This file can be unpacked via WinZip 8.x (or greater) for PC distributions.  Note earlier versions of WinZip (e.g., 7.0) do not extract zero-length files from archives.  Since XDC uses zero-length files to maintain timestamps, extracting XDC using an earlier version of WinZip will result in a corrupted installation.

If you have a UNIX distribution, use gunzip and tar to unpack the XDC distribution.

gunzip xdc-linux.tar.gz
tar xf xdc-linux.tar

When unpacked, this distribution consists of a single top-level directory that contains everything necessary to create and use XDC packages (with the exception of your code generation tools).  In the remainder of this note, we refer to this directory symbolically as xdcroot. 

Configuring Your Command Line Shell

The only command necessary to use the XDC tools is the xdc command.  This command must either be found along your PATH or you must define an alias that explicitly names the full path to the xdc command.  The table below illustrates how to modify your path to use the XDC tools.

Windows cmd.exe

Unix csh

Unix ksh

set PATH=%PATH%;xdcroot

setenv PATH="$PATH:xdcroot"

export PATH="$PATH:xdcroot"

Alternatively, one may define an alias that simply specifies the full path to the xdc command.  This is illustrated below.

Windows cmd.exe

Unix csh

Unix ksh

doskey xdc=xdcroot/xdc.exe $*

alias xdc xdcroot/xdc

alias xdc="xdcroot/xdc"

 

Configuring External Compilation Tools

Before the xdc command can generate a package’s makefile and compile it's sources, it must be told which targets should be used and, for each target, where its compiler is installed.  This information is specified in a “config.bld” file. 

Unless you explicitly tell the xdc command where to find this file on the command line or it’s in the current package directory, xdc will use the file named by the environment variable XDCBUILDCFG; see xdc.bld.BuildEnvironment for a complete description of the search algorithm.  For the purpose of this introduction, we'll simply set the XDCBUILDCFG environment variable.  For example from a Windows cmd.exe command shell:

set XDCBUILDCMD=c:/my/home/config.bld

The contents of the config.bld file – written in JavaScript – specify which target tool-chain to use and where it’s installed.  The example below specifies the TI C62 compiler, but all other compilers can be similarly configured.

var C62 = xdc.useModule("ti.targets.C62");

C62.rootDir = "C:/CCStudio_v3.3/C6000/cgtools";

Build.targets = [C62];

Testing the Installation

To ensure that everything is working properly, type the following commands:

cd xdcroot
xdc -PR examples/basic

The last command builds all of the basic example packages for all configured targets, and should complete without errors.

Hello World Example

The directory xdcroot/examples/basic contains several sample packages, which can be copied to your workspace; although you may develop and build within the distribution directory, it is good practice to maintain a separate working directory to avoid overwriting your work when updating to a new release of XDC. The package xdcroot/examples/basic/hello is a configurable form of the ever-popular hello world example.  We use this package to demonstrate how to develop and deliver XDC packages.

Although we use this package to demonstrate the mechanics of package management, the process is the same for all packages.  In particular, if you are interested in seeing how to encapsulate an existing code base as a package without making any changes to the C sources, the xdcroot/examples/legacy/hello package would be a good place to start.  This package encapsulates the hello world example from Kernighan and Ritchie’s text “The C Programming Language”.

Copying a Package

We begin by copying the basic.hello package to a working directory that is outside of xdcroot:

mkdir basic
cp -rp xdcroot/examples/basic/hello basic
cd basic/hello

Note that the name of the package directory is important: all XDC packages have names that must match the name of the directory holding the contents of the package.  In this case, the name of the package is “basic.hello” so the name of the directory containing the package must have a name that ends with “basic/hello”.  The name and version of a package is declared in the file package.xdc which is contained at the top of the package’s directory tree.

Cleaning a Package

In order to more easily understand the files necessary to create a package, we begin by removing all generated files. The following command will remove all files that are generated as part of building the package, leaving only the package’s source files.

xdc clean

The following table summarizes the source files that are part of the Hello package.

Description

Hello module’s definition; this file defines the C/C++ interface and the configuration interface to Hello module

Hello.c

Hello.cfg

To build the basic.hello package, type the command:

Testing a Package

To test the basic.hello package, type the command:

xdc test

The package’s build script can define one or more tests. A test is defined by an executable and a specified set of arguments (passed to the main() method’s argv[] parameter). The build script may optionally define an “exec” program that is used to load and execute the test’s program.

Building “test” causes all tests defined by the package’s build script to be run.

Managing Package Dependencies

In general, any package may depend on zero or more “external” packages in order to build or run.  Dependent packages must be located either along a “package repository” path defined by the variable XDCPATH, in the xdcroot/packages repository, or in the repository containing the current package; xdcroot/packages and the repository containing the current package are always searched after any repositories named by XDCPATH.

Since the basic.hello package only requires packages from the xdcroot/packages repository, there is no need to define XDCPATH in order to build basic.hello.  However, a client of the basic.hello package will need to modify XDCPATH to include the directory containing the basic.hello package directory (unless the basic.hello package and the client package are in the same repository.)  If, for example, the basic.hello package is installed in the repository c:/packages, a client can build using this package as follows:

xdc XDCPATH="c:/packages" all

As an alternative to defining XDCPATH on the command-line, an environment variable of the same name can be defined instead.

set XDCPATH="c:/packages"

If the XDCPATH environment variable is defined and XDCPATH is specified on the command line, the command line definition overrides the environment variable setting.

Delivering and Installing a Package

To deliver the basic.hello package to a consumer (i.e., someone who uses but does not modify the contents of the package), simply send them the file basic_hello.tar.  This file is automatically created as part of building the release goal in the basic.hello package.

xdc release
cp basic_hello.tar …

The package’s build script can define additional files that should be included in the tar file.  For example, platform packages such as ti.platforms.sim62xx include linker command files.

To install a package, simply unpack the tar file in any directory named in XDCPATH, say c:\packages.

cp basic_hello.tar c:\packages
cd c:\packages
tar xf basic_hello.tar
rm basic_hello.tar

The basic.client example shows how a consumer can use the basic.hello package installed along the XDCPATH.

Advanced Topics

Build Scripts

Build scripts operate on objects defined using the XDC Interface Definition Language.  In particular, the modules contained in the xdc.bld package located in xdcroot/packages define the build object model’s objects.

The module xdc.bld.BuildEnvironment provides “global” context read by build scripts.  The xdc.bld.PackageContents module, on the other hand, is modified by the build script by populating the package set with the physical files (programs, libraries, object files, documentation, etc.) which makeup the delivered package.

Although the names “xdc.bld.BuildEnvironment” and “xdc.bld.PackageContents” are descriptive, they are somewhat long and can make build scripts hard to read.  Therefore, standard short aliases for these modules exist at the time build scripts run; “Build” is an alias for the xdc.bld.BuildEnvironment module, and “Pkg” is an alias for the xdc.bld.PackageContents module.

Debugging Build Scripts

Build scripts are extremely powerful but this power can make the scripts difficult to read and understand.  Of course, one can always use “printf” style debugging within a build script; all objects support the ability to display themselves as a string and any XDC script can use the “print” function to display a string.

In addition, as part of the XDC makefile generation process, a hierarchical XML representation of the build goals is also produced (package/package.bld.xml).  Using a GUI XML reader, it is possible to easily review the results of the build script and determine whether the appropriate build goals and options are specified.

Finally, since build scripts are run via the tconf tool and this tool has a built-in graphical debugger, it is even possible to single-step debug build scripts.  Recall that the xdc command is simply a “wrapper” for GNU make, so we need a way to tell make to pass the “-g” option to tconf whenever tconf is used to run a build script.  This can be done by either setting the environment variable MK_BLDOPTS to “-g” or by directly setting this variable on the xdc command line.

xdc MK_BLDOPTS=-g …

Targets and Platforms

In XDC, a Target encapsulates all the information necessary to build an object file and archive it in a library.  Intuitively a target corresponds to a CPU Instruction Set Architecture (ISA) that is the target instruction set of the compiler together with one of the compiler’s runtime models (big endian, little endian, near, far, …).  A platform, on the other hand, encapsulates the information necessary to create and run executables.  Platforms include memory map information, external peripheral information, one or more CPUs and their revision numbers, as well as one or more boards and their revision numbers.

Executables are built to run on platforms and an executable can run on just one CPU of a particular platform.  Libraries and object files are built for a target and can be used by any executable that runs on a platform that supports the specified target.  Libraries and objects built for the same target may always be compatibly linked together into a single executable.

Build and Configuration Domains

The XDC development environment has a number of scripts that are used to both specify the elements of a package as well control the design-time configuration of individual programs.  The build script, package.bld, populates the Build Object Model (BOM) whereas program configuration scripts, *.cfg, populate the Target Content Object Model (TCOM).  It is important to realize that while there are many similarities between these models, they are distinct “domains”; i.e., the objects in the BOM have distinct properties and methods that are separate from the objects in the TCOM.

The BOM includes information about collections of libraries, programs, and headers that make up a package; it includes everything necessary to build the package.  TCOM, on the other hand, includes information about a single program that is a part of a package; e.g., specific configurations of modules included in the program.

It is often desirable to configure a program conditionally as a function of attributes specified in the BOM.  For example, one may want to select a “debug” configuration of a package as a function of the attributes specified in the BOM.

Adding a New Target

This section describes how to add a new target to the XDC build system.

Adding a new target requires the creation of a module that implements the xdc.bld.ITarget interface.  The global build configuration script (e.g., config.bld) must load the package containing this module and simply add this module to the Build.targets array.

Since host-side interfaces may contain default implementations of methods, it is often possible to “inherit” an existing implementation of the xdc.bld.ITarget interface from another target.  For example, when adding a new target that uses a TI compiler, it is often possible to simply create a new module that extends ti.targets.ITarget.  In this case, one can add a target by simply initializing data parameters specified in this interface; the implementation reads these parameters and performs the appropriate actions.

Re-Building Target Runtime Support

There are situations where it is necessary to rebuild the runtime support libraries that are delivered with a package of targets.  This usually occurs because the existing runtime support libraries have been built with a newer compiler than you are using.  If it is not possible to upgrade to a newer compiler, it is possible to rebuild the runtime support libraries as follows:

cd targets-package
xdc -PR .

For example, if your host development environment is a PC running Linux or Windows and you have to build native executables with your existing compiler, you can build the XDC runtime support libraries for your target.

For Linux workstations, type:

cd xdcroot/packages/gnu/targets
xdc -PR .

For Windows workstations, type:

cd xdcroot/packages/microsoft/targets
xdc -PR .

These commands will build the XDC runtime support libraries for all configured native targets, and should complete without errors.

Adding a New Platform

This section describes how to add a new platform to the XDC build system.

Adding a new platform to XDC requires the creation of a “platform package”.  A platform package is like any other XDC package except that it must contain a module named “Platform” that implements the xdc.platform.IPlatform interface.

The Package Path and XDCPATH

This section describes how to use the XDC Package Path to control the packages that are part of a development environment.  The package path names one or more directories that contain package directories.  This allows packages to be installed anywhere on a system.  It also allows one to work with multiple versions of the same package.

When building or using a package, the XDC tools will search for a package’s files by looking for the package’s directory in the directories specified by the path

$(XDCPATH);$(XDCROOT)/packages;^

where $(XDCPATH) is the value of the optionally defined XDCPATH environment variable, $(XDCROOT) is the installation directory of the xdc command, and ^ is the relative directory path to the repository containing the package being built.  For example, when building a package named “ti.bios”, the symbol ^ is replaced with “../../” but for a package named “hello” the symbol ^ is replaced with “../”.  By having this directory or “repository” in the package path, it is possible to work within a package that depends on other packages contained in this repository without ever having to set the XDCPATH environment variable.

The package path does not name package directories; it names repositories (i.e., directories containing package directories).  Suppose, for example, that all packages from third parties are installed in /usr/local/packages and you are also developing several packages of you own (located in /usr/home/me).  The following setting of the XDCPATH will allow your packages to use any package in /usr/local/packages or your own packages.

set XDCPATH=/usr/local/packages;/usr/home/me

Note that directories are always separated by the ‘;’ character; even on Unix based development systems. 

It is usually a mistake to put a relative path in the XDCPATH environment variable.  Relative paths in XDCPATH reference directories relative to the package being built rather than the directory where the xdc command was invoked.  Thus, a relative path will refer to a different repository for each package being built. 

It is possible, however, to use the ‘^’character in the XDCPATH definition to refer to the current package’s repository.  So, if you have a repository that is always in a fixed location relative to all of your packages repositories, it is possible to create a single XDCPATH setting that does not include any absolute paths.  Suppose, for example, that your build system places all prerequisite packages in an “imports” repository prior to building the packages in a “src” repository and the import and src repositories are sibling directories in the file system.  The following XDCPATH setting is sufficient to build all packages in the src repository.

set XDCPATH=^/../imports

Note that multiple versions of the same package can appear along the XDCPATH.  The package path can name multiple “package repositories” which can contain a package directory with the same name.  When searching for a package, the first repository that contains a directory matching the package’s name will be used.  Thus, even if two packages with the same name appear in the package path, only one will ever be found; i.e., the first one in the order specified in the package path.

Package Names

All modules and interfaces are part of a named package.  If the name is non-empty, the name must “match” a suffix of the current working directory name; each component of the package name must match the name of a corresponding sub-directory name.  For example, the package name “examples.basic.hello” must be located in a directory that ends with “examples/basic/hello”.  All module names begin with the package name followed by the dot (‘.’) character. 

If a package’s name is the empty name, the last sub-directory name of the current working directory is used for locating libraries and interfaces.  However, all module and interface names are not enclosed in any namespace.  Thus, clients of the module Mod in an unnamed package in the directory dir must #include <dir/Mod.h> in C/C++ source files and link with libraries named dir/lib (where dir/lib is the library name returned by the unnamed package’s getLibs() method).  Since the package name is empty, the client can directly refer to the module Mod without any package namespace qualifier.

It is possible to use “#include <Mod.h>” in-lieu of “#include <dir/Mod.h>” (thereby eliminating the package directory names from the C/C++ files that refer to a package) by specifying in the package.bld build script that the referring package “uses” dir.

Pkg.uses = ["bios/legacy"];

The package path will be searched for a directory that matches the path suffixes named and these directories are added to the C/C++ compiler’s include paths.

The xdc Command

The xdc command is used to build packages and programs that use packages.  In its current implementation, xdc is nothing more than command shell that invokes the GNU make utility with makefiles that are either part of the XDC toolset or generated as part of processing the package build script (i.e., package.bld).

Complete documentation of this command together with examples is part of the xdc man page.

Known Problems

This section contains a description of known problems with the current release.

·        The current dependency generation is much too conservative; in many situations we unnecessarily re-build files.  For example, executables will re-configure and re-link if a package it depends on has been cleanly rebuilt.  This is a consequence of how we determine changes to an interface.

·        The package path allows multiple packages with the same name to be part of the build environment.  This can be a useful when building a “replacement” package for a package that may appear in a central read-only repository together with other packages that are needed.  In practice, however, this technique may result in build errors because of an improper setting of the package path: an unintended version of the duplicate package is used.

·        If package B depends on package A, installation of a different version of package A should cause the appropriate files in package B to be re-built.  However, because the underlying build engine is GNU make, package B will re-build only if the files in package A are newer; i.e., replacing a package with an older version of itself does not necessarily trigger a re-build.
